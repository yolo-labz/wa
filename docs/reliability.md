# Reliability Principles for Speckit + Claude Code

This document is the long-form rationale and citation trail for the binding rules summarised in [`CLAUDE.md`](../CLAUDE.md) §"Reliability principles". It synthesises five research dossiers run by a parallel agent swarm on 2026-04-06:

- [`research-dossiers/speckit-quality.md`](./research-dossiers/speckit-quality.md) — official guidance, community failure modes, comparison with other agent rule systems
- [`research-dossiers/spec-driven-theory.md`](./research-dossiers/spec-driven-theory.md) — Hillel Wayne, Lamport, Hoare, Liskov, Ousterhout, IEEE 830, Living Documentation
- [`research-dossiers/llm-coding-reliability.md`](./research-dossiers/llm-coding-reliability.md) — Anthropic context engineering, lost-in-the-middle, hallucination triggers, sycophancy, scope creep
- [`research-dossiers/architecture-quality-measurement.md`](./research-dossiers/architecture-quality-measurement.md) — ATAM, ARID, ADRs, C4, Keeling, Hohpe, Cockburn original
- [`research-dossiers/hexagonal-antipatterns.md`](./research-dossiers/hexagonal-antipatterns.md) — Vernon, Evans, Martin, ThreeDotsLabs, Cheney, Ryer, Bourgon — 13 PR-review anti-patterns

**The thesis**: spec-driven development with an LLM coding agent works only if the spec is *load-bearing* — written precisely enough that the agent cannot satisfy it by hallucinating, ignored only at the cost of immediate failure, and updated by re-running the spec command, never by hand-editing artefacts. Every rule below is a defence against a documented failure mode.

## A. Speckit workflow discipline

**A1. Constitution-first.** `.specify/memory/constitution.md` MUST exist with versioned, falsifiable principles before the first `/speckit:specify`. "We value quality" is not a principle; "all HTTP handlers return `(T, error)`" is. Aspirational constitutions are ignored by both humans and agents.
*Why*: r/ClaudeAI "dead constitution" reports; spec-kit `docs/spec-driven.md`. *Source*: [speckit-quality §3, §7].

**A2. No hand-edits to generated artefacts.** `spec.md`, `plan.md`, `tasks.md`, and `research.md` are regenerated by re-running their respective slash commands, never patched in place. The only exception is the manual fill that the slash command itself prompts you to do during its first run. Hand-edits create the "spec laundering" anti-pattern: the agent edits the spec to match the code it just wrote rather than the reverse.
*Why*: spec-kit `docs/spec-driven.md`; observed failure mode in r/ClaudeAI postmortems. *Source*: [speckit-quality §1, §7].

**A3. `/clarify` before `/plan`, always.** No `/plan` invocation may run while `[NEEDS CLARIFICATION]` markers remain in the spec. Hand-waving Phase 0 with "TBD" produces architecture-by-accident in `/plan`.
*Why*: spec-kit v0.4.x release notes added `/clarify` for this exact reason. *Source*: [speckit-quality §1, §7].

**A4. `/analyze` before `/implement` from feature 002 onward.** `/analyze` runs a non-destructive cross-artefact consistency check across `spec.md`, `plan.md`, and `tasks.md`. From the second feature onward, multi-feature drift becomes possible — feature 003 might silently violate feature 001's constitution. `/analyze` catches it before code lands.
*Why*: spec-kit multi-feature drift issues in upstream repo. *Source*: [speckit-quality §3, §5].

**A5. `data-model.md` is the single field authority.** `/implement` may not reference any entity, field, or type that does not appear in the active feature's `data-model.md`. If a field is missing, stop and run `/speckit:plan` again with the addition; do not invent.
*Why*: the most-cited LLM hallucination failure mode in production speckit reports. *Source*: [speckit-quality §3]; [llm-coding-reliability §3, §7].

**A6. One feature in flight per branch; cap `tasks.md` at ~25 items.** Feature granularity should equal one PR's worth of work. If `/speckit:tasks` produces more than ~25 items, split the feature. Multi-feature parallelism multiplies drift risk superlinearly.
*Why*: community drift reports; spec-kit maintainer guidance. *Source*: [speckit-quality §3, §5].

**A7. Constitution amendments require a version bump and a migration note.** Every change to `.specify/memory/constitution.md` increments `Version:` and includes a paragraph in the PR explaining which principles were touched and how downstream features must adapt. A constitution that drifts silently is a constitution that gets ignored.
*Why*: spec-kit `docs/spec-driven.md`. *Source*: [speckit-quality §1, §4].

## B. Spec quality (the IEEE 830 / Wayne / Lamport rules)

**B1. Every requirement must be verifiable by a finite check.** No adjectives without thresholds. "Fast", "user-friendly", "robust" are forbidden. Every requirement must have a corresponding pass/fail observation a third party can run.
*Why*: IEEE 830-1998 §4.3.6; ISO/IEC/IEEE 29148:2018. *Source*: [spec-driven-theory §9].

**B2. Specify what, not how.** Port-interface specs describe observable behaviour at the boundary; implementation details live in code. Mixing the two couples clients to implementation choices that should be free to change.
*Why*: Parnas 1972 ("On the Criteria To Be Used in Decomposing Systems into Modules"); Liskov 1974. *Source*: [spec-driven-theory §3, §4, §8].

**B3. The interface must be simpler than the implementation it hides.** If the spec for a port grows faster than the code that implements it, the module is shallow — redesign. Apply Ousterhout's deep-module ratio to every port: interface complexity / implementation complexity should be ≪ 1.
*Why*: Ousterhout, *A Philosophy of Software Design* ch. 4. *Source*: [spec-driven-theory §5]; [architecture-quality-measurement §6].

**B4. Pair every behavioural claim with a concrete example AND a universal property.** Examples (Given/When/Then) prevent ambiguity; properties prevent overfitting. A spec that has only examples produces an implementation that handles only those exact cases.
*Why*: Wayne, *Why Don't People Use Formal Methods?*; Adzic, *Specification by Example*. *Source*: [spec-driven-theory §1, §8].

**B5. Define errors out of existence at the boundary.** Narrow input types until illegal states are unrepresentable. The strongest API is the one where the error case cannot be expressed by a caller with valid types.
*Why*: Ousterhout ch. 10 ("Define errors out of existence"). *Source*: [spec-driven-theory §5].

**B6. State every assumption a client may rely on; everything else is unstable.** Unwritten assumptions become accidental load-bearing API. Every "naturally" or "obviously" or "of course" in a spec is a hidden assumption that should be promoted to a numbered rule or removed.
*Why*: Parnas 1972 — "the connections between modules are the assumptions which the modules make about each other." *Source*: [spec-driven-theory §8].

**B7. The spec must be checked in CI against the code.** Unverified specs become folklore. For this project, the depguard rule, the `forbidigo` rule, and the contract test suite collectively form the spec-to-code consistency check.
*Why*: Martraire, *Living Documentation*. *Source*: [spec-driven-theory §8].

**B8. Subtype/implementation substitution must preserve the spec's pre/postconditions, not just its types.** LSP at the prose level: every adapter satisfying a port must satisfy not just the Go interface but every behavioural clause in the contract document.
*Why*: Liskov 1988 ("Data Abstraction and Hierarchy"); Hoare 1969. *Source*: [spec-driven-theory §3, §4].

## C. LLM coding agent reliability (the Liu / Anthropic / METR rules)

**C1. Read before you write.** Before any `Edit` or `Write` to path P, the agent must `Read` P (or confirm P does not exist via `Glob`). Pre-task grounding is the single most reliable hallucination suppressant per Jimenez et al. SWE-bench 2024.
*Why*: Anthropic Claude Code best practices; SWE-bench retrieval ablation. *Source*: [llm-coding-reliability §3, §5].

**C2. Cite `file:line` for every factual claim about this codebase.** Claims without a path:line citation are prohibited. Citation-required prompting is a documented hallucination suppressant.
*Why*: Liu 2023 lost-in-the-middle; reduces hallucination per CodeHallu 2024. *Source*: [llm-coding-reliability §2, §3, §5].

**C3. No silent fallbacks.** Never wrap an error in a default-returning try/except unless the spec explicitly calls for it. Surface errors visibly. The "helpful mocking" anti-pattern — agent stubs out a failing dependency and marks the task done — is forbidden.
*Why*: Anthropic Claude Code best practices, April 2025. *Source*: [llm-coding-reliability §4].

**C4. No scope creep.** Touch only files named in the active `tasks.md` item. Unrelated fixes go in a new task, not this diff. The "while I'm here..." pattern is forbidden.
*Why*: Cognition "Don't Build Multi-Agents"; SWE-bench error taxonomy. *Source*: [llm-coding-reliability §4].

**C5. Negations are prohibitions, not examples.** If the spec says "do not use X", X must not appear in the diff. LLMs systematically under-weight "not" by 30-60% (Truong et al. 2023) — re-read the spec before committing and ask "does my diff contain anything the spec forbids?"
*Why*: Truong et al. 2023, "Language Models Are Not Naysayers". *Source*: [llm-coding-reliability §4].

**C6. Plan mode first for any task touching more than 2 files.** Produce a plan that grep-verifies each referenced symbol exists before leaving plan mode. This is the Claude Code analogue of the read-before-write rule scaled up to multi-file operations.
*Why*: Anthropic plan-mode docs. *Source*: [llm-coding-reliability §1, §6].

**C7. Tests run, or the task is not done.** `[x]` in `tasks.md` requires a passing test referenced by name. The "deadline-framing test-skip" failure mode (METR 2025) is mitigated by treating the test pass as the gate, not the agent's self-assessment.
*Why*: METR 2025 agent evals. *Source*: [llm-coding-reliability §4, §5].

**C8. Never edit `spec.md`, `plan.md`, or `constitution.md` from `/implement`.** Spec edits require an explicit `/specify` or `/plan` invocation. The "spec laundering" anti-pattern — agent rewrites the spec to match what it just coded — is forbidden by construction.
*Why*: speckit workflow; observed failure in r/ClaudeAI. *Source*: [llm-coding-reliability §7]; [speckit-quality §3].

**C9. Challenge wrong premises.** If the user's request contradicts the spec, the constitution, or a file the agent just read, the agent must say so before acting. Sycophancy ("yes, and...") leads to compounding errors.
*Why*: Sharma et al. 2023 ("Towards Understanding Sycophancy in Language Models"). *Source*: [llm-coding-reliability §4].

**C10. Keep CLAUDE.md under 400 lines.** Rules past line ~400 measurably stop firing because of lost-in-the-middle attention degradation. Long-form rationale lives in `docs/reliability.md` (this file) and is `@import`-referenced. The CLAUDE.md proper holds only the binding rules.
*Why*: Liu 2023; RULER 2024 (Hsieh et al.); NoLiMa 2025 (Modarressi et al.). *Source*: [llm-coding-reliability §2].

**C11. Dispatch a subagent for any read exceeding ~20K tokens.** Large log dumps, vendored code, generated files — return only the extracted answer to the parent. Subagent dispatch isolates context and prevents context rot.
*Why*: Anthropic "Effective context engineering for AI agents", Sept 2025. *Source*: [llm-coding-reliability §1].

## D. Architectural decision quality (the ATAM / ADR / Cockburn rules)

**D1. Every architectural decision in `research.md` MUST name at least one rejected alternative with its reason.** A decision without alternatives is a decision log, not a decision record. MADR-style ADR completeness is the bar.
*Why*: Nygard 2011, "Documenting Architecture Decisions"; MADR template. *Source*: [architecture-quality-measurement §2].

**D2. Every port name describes an *intent of conversation*, not a technology or external system.** Bad port names: `DatabasePort`, `KafkaPort`, `WhatsAppPort`. Good port names: `SessionStore`, `EventStream`, `Allowlist`. **The number of ports is determined by the number of distinct conversations, not by a fixed convention** — Cockburn's original 2005 paper explicitly says *"the number six is not important... it is a symbol for the drawing, and leaves room to insert ports and adapters as needed."*
*Why*: Cockburn 2005, "Hexagonal Architecture". *Source*: [hexagonal-antipatterns §1, §11]; [architecture-quality-measurement §7].

**D3. The port set is COMPLETE if every use case is expressible using only the declared ports AND every port is used by at least one use case.** Over-decomposition shows as ports with one method, one implementation, one caller — collapse them. Under-decomposition shows as a port whose methods have unrelated callers — split along bounded-context lines.
*Why*: Cockburn 2005 completeness test; Vernon IDDD ch. 2. *Source*: [hexagonal-antipatterns §11]; [architecture-quality-measurement §7].

**D4. Each top engineering risk MUST be addressed by a specific decision in the spec.** Risk-driven design: do as much architecture as the risk demands, no more. If a risk has no corresponding decision, the spec is incomplete.
*Why*: Keeling, *Design It!* 2017; Fairbanks, *Just Enough Software Architecture* 2010. *Source*: [architecture-quality-measurement §4].

**D5. The spec must contain at least one quality-attribute scenario per ATAM dimension** (performance, availability, security, modifiability) **with a measurable response**. Happy-path-only specs predict happy-path-only implementations.
*Why*: Kazman/Bass/Klein, ATAM 2000; ISO/IEC 25010. *Source*: [architecture-quality-measurement §1, §10].

**D6. No infrastructure types in port signatures.** No SQL, HTTP, framework, serialisation, or library types in the seven port interfaces. The depguard rule `core-no-whatsmeow` is the mechanical enforcement; this is the human-readable principle.
*Why*: Vernon IDDD; Spolsky 2002 "Leaky Abstractions". *Source*: [hexagonal-antipatterns §2, §3]; [architecture-quality-measurement §9].

**D7. Domain invariants must be encoded as types or tests, not prose.** "The body must not exceed 64KB" in a spec must correspond to either a type that cannot represent a >64KB body or a test that asserts the limit. Prose-only invariants drift.
*Why*: Evans, *DDD* 2003; Wayne, *Practical TLA+* 2018. *Source*: [architecture-quality-measurement §9].

## E. Hexagonal anti-patterns (the Vernon / Cheney / Ryer / Bourgon rules)

**E1. Port interfaces live in the consumer package (the domain or use case), not in the producer package.** Cheney's "the bigger the interface, the weaker the abstraction" applies: declare the interface where it is *used*, not where it is implemented.
*Why*: Cheney 2016, "SOLID Go Design". *Source*: [hexagonal-antipatterns §7].

**E2. Do not declare an interface before a second implementation or a real test double exists.** Interface-up-front is a smell. The exception is when the interface is a port boundary specifically designed for substitution — which is exactly this project's case, and which is justified inline in `data-model.md`.
*Why*: Ryer GopherCon; Cheney. *Source*: [hexagonal-antipatterns §6, §7].

**E3. No `Repository` named after a SQL table.** Repositories correspond to aggregate lifecycle (`Save`, `ByID`, `Remove`), not to one-method-per-query CRUD. This project deliberately avoids the `Repository` name for `SessionStore`, `ContactDirectory`, `GroupManager` to escape the CRUD-in-DDD-costume trap.
*Why*: Vernon IDDD ch. 12; ThreeDotsLabs "Repository Pattern". *Source*: [hexagonal-antipatterns §2, §5].

**E4. No anemic structs.** Every exported domain type has at least one method beyond accessors. `Get` and `Set` pairs with all logic in a `Service` are forbidden.
*Why*: Fowler 2003 "AnemicDomainModel" via Vernon IDDD ch. 1. *Source*: [hexagonal-antipatterns §2, §5].

**E5. No business rules in HTTP handlers, gRPC servers, or unix-socket handlers.** Handlers unmarshal, call a use-case function, marshal the response. Nothing else.
*Why*: ThreeDotsLabs "Clean Architecture in Go". *Source*: [hexagonal-antipatterns §5].

**E6. No DTOs at every layer boundary.** One DTO at the transport edge; domain types cross internal boundaries.
*Why*: Bourgon "Go Best Practices". *Source*: [hexagonal-antipatterns §8].

**E7. No `FooService` / `FooManager` / `FooHandler` / `FooInteractor` for the same concept.** One name per concept, chosen from the ubiquitous language. Service/Manager/Handler suffix proliferation is a structural smell.
*Why*: Ryer GopherCon; Cockburn on layer asymmetry. *Source*: [hexagonal-antipatterns §6, §9].

**E8. Wire dependencies explicitly in `main.go` (or a small `service` package).** Do not use a reflective DI container. `wire` (compile-time) is acceptable; `dig`-style reflective containers are not.
*Why*: Bourgon; Ryer. *Source*: [hexagonal-antipatterns §8, §11].

**E9. Tests use in-memory adapters, not mock-everything frameworks.** Mocks coupled to call order are forbidden. The contract test suite under `internal/app/porttest/` is this project's enforcement.
*Why*: ThreeDotsLabs "Tests in Go"; Vernon IDDD ch. 4. *Source*: [hexagonal-antipatterns §12].

**E10. A "use case" that is a one-line delegation to a single repository call is a port in disguise — collapse it or call the port directly.** Use cases earn their existence by orchestrating multiple ports or applying domain logic.
*Why*: Martin, *Clean Architecture* ch. 20. *Source*: [hexagonal-antipatterns §13].

## How this document is enforced

Most of the rules above are advisory at the natural-language level. The mechanically-enforced subset is:

| Rule | Mechanism |
|---|---|
| A1, A7 | Constitution version field; PR review |
| A5 | `/speckit:analyze` before `/implement` |
| B1, B2, B5 | Spec quality checklists (`requirements.md`, `architecture.md`) |
| B7 | `core-no-whatsmeow` `depguard` rule + `forbidigo` + the contract test suite |
| C1 | Built-in Claude Code soft rule + plan mode |
| C2 | PR review |
| C5 | `forbidigo` rule that bans forbidden patterns |
| C7 | `lefthook` `pre-push` hook running `go test` + `govulncheck` |
| C10 | `wc -l CLAUDE.md` in the `quickstart.md` of every feature |
| D6 | `core-no-whatsmeow` `depguard` rule |
| E1, E2, E5, E7, E8 | PR review against this document |
| E4 | `golangci-lint` `revive` rule + PR review |
| E9 | `internal/adapters/secondary/memory` package and the contract suite |

The unmechanised rules rely on pre-commit habit and PR review. As the project grows, more should migrate from "PR review" to "linter" to "type system".
